1. What is a Sorting Network?

A sorting network is a fixed, data-independent network of comparators. Unlike a conventional sorting algorithm (like quicksort or merge sort), a sorting network's structure is determined entirely ahead of time — it doesn't branch based on the values it's sorting. The same sequence of comparisons is applied to every input.

This makes sorting networks ideal for hardware implementations (like FPGAs and GPUs), where you want to exploit massive parallelism. Every comparator that doesn't depend on another can fire simultaneously.

Key Concept: Data Independence A sorting network compares the same pairs in the same order no matter what values are in the input. This is what makes it a "network" rather than an "algorithm" — it's a circuit, not a program.

2. Comparators & the Compare-Swap Operation

The only operation in a sorting network is the comparator: it takes two wires, compares their values, and places the smaller on the top wire and the larger on the bottom wire. If they're already in order, nothing changes.

COMPARE-SWAP(A, i, j):     // i < j
    if A[i] > A[j]:
        swap A[i] and A[j]
// After: A[i] ≤ A[j] — always

Comparators are drawn as vertical bars connecting two horizontal "wires." Values flow from left to right. A comparator bar means "ensure the smaller value is on the upper wire."

Depth vs Comparators: Two metrics matter. Depth = the length of the longest path of dependent comparators (this determines latency in parallel hardware). Comparators = total number of compare-swap operations (this determines hardware cost). We want both to be small.

3. Batcher's Odd-Even Merge (1968)

Kenneth Batcher invented the Odd-Even Merge Sort in 1968. The key insight is a recursive merge procedure that works on two sorted halves.

ODD-EVEN-MERGE(A[0..n-1]): To merge two sorted sequences of length n/2 each:

ODD-EVEN-MERGE(lo, hi, step):
    half = step * 2
    if half < hi - lo:
        ODD-EVEN-MERGE(lo, hi, half)       // merge odd-indexed elements
        ODD-EVEN-MERGE(lo + step, hi, half) // merge even-indexed elements
        for i = lo + step to hi - step, step 2*step:
            COMPARE-SWAP(i, i + step)       // final compare-swap layer
    else:
        COMPARE-SWAP(lo, lo + step)         // base case
The Odd-Even Split Insight Split both sorted halves into their odd-indexed and even-indexed elements. Recursively merge the odd halves together, and the even halves together. The resulting sequences are "almost sorted" — they only need one additional layer of compare-swap operations on adjacent pairs to become fully sorted.

4. Recursive Structure — How the Full Sort Works

The full ODD-EVEN-MERGE-SORT applies the merge recursively to sort the entire array:

ODD-EVEN-SORT(A, lo, hi):
    if hi - lo > 1:
        mid = (lo + hi) / 2
        ODD-EVEN-SORT(A, lo, mid)    // sort first half
        ODD-EVEN-SORT(A, mid, hi)    // sort second half
        ODD-EVEN-MERGE(A, lo, hi, 1) // merge two sorted halves

The recursion bottom-out when subproblems have size 1 (trivially sorted). The visualization color-codes odd-indexed wires and even-indexed wires so you can see the split structure as it animates.

Worked Example — 4 elements [3, 1, 4, 2]:
Step 1: Sort [3,1] → [1,3]   Step 2: Sort [4,2] → [2,4]
Now merge [1,3] and [2,4] using odd-even merge:
— Odd sub-merge: [1] and [2] → [1,2]
— Even sub-merge: [3] and [4] → [3,4]
— Interleave: [1,3,2,4]   — Final compare-swap: [1,2,3,4] ✓

5. The 0-1 Principle — Proving Correctness

How do we prove a sorting network sorts all inputs correctly without testing every possible permutation? The 0-1 Principle gives us a shortcut:

0-1 Principle (Knuth, 1973) A comparison network correctly sorts all inputs of arbitrary values if and only if it correctly sorts all binary (0/1) inputs. There are only 2n binary inputs — far fewer than n! permutations!

The proof relies on the fact that any monotone function f (like "≥ threshold") applied element-wise to the input produces a consistently mapped output. If the network sorts all binary sequences, it must sort all sequences of any comparable values.

In the visualization, you can activate 0-1 Test Mode to cycle through all binary inputs and verify the network sorts each one correctly.

7. Bitonic Sort vs Odd-Even Sort

Bitonic Sort works by constructing a bitonic sequence (first increasing then decreasing, or vice versa) and then sorting it. It requires the input size to be a power of 2 and generates comparator patterns with a distinctive "butterfly" shape.

Odd-Even Merge Sort also requires power-of-2 sizes, but uses the odd/even split merge strategy. The two algorithms produce different comparator diagrams and different intermediate sorted states, even though their asymptotic complexity is identical.

📖 Lecture Notes — FFT: Polynomial Multiplication  Start Here

1. The Polynomial Multiplication Problem

Given two polynomials A(x) = a₀ + a₁x + a₂x² + … + aₙ₋₁xⁿ⁻¹ and B(x) = b₀ + b₁x + … + bₙ₋₁xⁿ⁻¹, compute their product C(x) = A(x) · B(x).

The result C(x) has degree up to 2n−2 and its coefficients are given by the convolution: cₖ = Σ aᵢ · bₖ₋ᵢ. This operation appears everywhere: signal processing (convolution of two signals), number multiplication (integers are base-10 polynomials), image filtering, and cryptography.

Engineering Problem Framing (CLO-2): In real systems such as image convolution and audio denoising, one output frame may require thousands of polynomial-like convolutions. A quadratic-time method becomes the bottleneck, so we reformulate the problem as FFT evaluation and interpolation to meet practical runtime limits.
Why This Matters The naïve algorithm for computing cₖ requires O(n²) multiplications — computing each of the 2n coefficients of C requires up to n multiplications. For large n (e.g., n = 10⁶), this is 10¹² operations. FFT reduces this to O(n log n) — about 20 million operations. That's a 50,000× speedup.

2. Naïve O(n²) Coefficient Multiplication

NAIVE-MULTIPLY(A[0..n-1], B[0..n-1]):
    C = array of zeros of size 2n-1
    for i = 0 to n-1:
        for j = 0 to n-1:
            C[i+j] += A[i] * B[j]    // n² multiplications total
    return C

For each output coefficient C[k], we sum all products A[i]·B[k−i]. This is straightforward but requires n² multiplications and n² additions. The visualization shows this approach so you can compare the operation count directly with FFT.

3. Roots of Unity — The Key Ingredient

The n-th roots of unity are the n complex numbers ω⁰, ω¹, ω², …, ωⁿ⁻¹ where ω = e^(2πi/n) = cos(2π/n) + i·sin(2π/n). They are evenly spaced on the unit circle in the complex plane.

The Cancellation Lemma ω^(n/2) = −1, which means (ω^k)^(n/2) = (−1)^k. This causes massive cancellations when we split a DFT of size n into two DFTs of size n/2, because the twiddle factors for the second half are just the negatives of the first half.
The Halving Lemma The squares of the n-th roots of unity are exactly the n/2 roots of unity: {(ω^k)²} = {ω²^k} and ω² = e^(2πi·2/n) = e^(2πi/(n/2)). This means squaring the n twiddle factors gives exactly the n/2 twiddle factors for a problem of half the size — enabling the divide-and-conquer recursion.

The unit circle visualization shows these roots as colored dots. Watch them appear and notice the symmetry: roots on the left are the negatives of roots on the right (they are π apart).

4. The FFT Idea — Evaluate, Don't Convolve

Instead of multiplying polynomials coefficient-by-coefficient, FFT uses a different representation. A polynomial of degree n−1 is uniquely determined by its values at n points. So instead of convolving coefficients, we can:

Step 1 — EVALUATE:   Compute A(ωᵏ) and B(ωᵏ) for k=0..2n-1  [O(n log n)]
Step 2 — POINTWISE:  C(ωᵏ) = A(ωᵏ) · B(ωᵏ) for each k      [O(n)]
Step 3 — INTERPOLATE: Recover C's coefficients from its values [O(n log n)]

The magic is that step 1 and step 3 can both be done in O(n log n) using the FFT — which exploits the special structure of the roots of unity to avoid redundant work.

5. Butterfly Operations — The FFT Mechanism

The FFT uses divide and conquer: split polynomial A(x) into its even-indexed coefficients A_even(x²) = a₀ + a₂x² + a₄x⁴ + … and odd-indexed A_odd(x²) = a₁ + a₃x² + …, so A(x) = A_even(x²) + x·A_odd(x²).

At each level of the recursion, values from the two halves are combined using a butterfly operation:

BUTTERFLY(a, b, ω):           // ω = twiddle factor at this level
    t = ω · b
    upper = a + t             // "top" output
    lower = a - t             // "bottom" output
    return (upper, lower)
Why "Butterfly"? The data flow diagram looks like a butterfly: two inputs cross over, each output is a sum/difference of one original value and a twiddle-factor-scaled version of the other. The butterfly is the core O(1) operation; there are O(n log n) butterflies total.
The Full Recurrence: T(n) = 2T(n/2) + O(n). Each level of the recursion does O(n) butterfly operations, and there are log₂(n) levels, giving O(n log n) total work. The recurrence is identical to merge sort!

7. Inverse FFT

The IFFT recovers coefficients from point values. Remarkably, it is almost identical to the FFT — just replace ω with ω⁻¹ = e^(−2πi/n) and divide the final result by n:

IFFT(Y[0..n-1]):
    result = FFT(Y, use ω⁻¹ instead of ω)
    for k = 0 to n-1:
        result[k] = result[k] / n
    return result

This works because the DFT matrix is a Vandermonde matrix of ω values, and its inverse is obtained by conjugating ω and scaling by 1/n. The symmetry between FFT and IFFT is one of the most beautiful results in computational mathematics.

1. Why Primality Testing Matters

Primality testing is the foundation of modern cryptography. RSA encryption requires generating two large random primes p and q (typically 1024–4096 bits each). To generate them, we pick random odd numbers and test them for primality until we find one that passes.

Naïve trial division up to √n is completely impractical for 2048-bit numbers — √(2²⁰⁴⁸) ≈ 2¹⁰²⁴, which is more atoms than in the observable universe. We need a fundamentally different approach.

The Goal Given a large odd integer n, determine whether it is prime — fast. Miller-Rabin does this in O(k · log²n) time where k is the number of rounds. For k=40 rounds, the probability of a false positive is at most 4⁻⁴⁰ ≈ 10⁻²⁴ — effectively zero.

2. Fermat's Little Theorem

The basis for many primality tests is Fermat's Little Theorem: if p is prime and gcd(a, p) = 1, then a^(p−1) ≡ 1 (mod p). In other words, raising any non-multiple of p to the power p−1 gives 1 mod p.

The Fermat primality test: pick a random witness a, compute a^(n−1) mod n. If the result ≠ 1, then n is definitely composite. If it equals 1, then n is probably prime.

The Problem: Some composite numbers pass the Fermat test for every possible witness a with gcd(a,n)=1. These are called Carmichael numbers. The smallest is 561 = 3×11×17. No matter which witness you pick, 561 looks prime to Fermat's test — yet it's clearly composite!

3. Carmichael Numbers — Fermat's Fatal Weakness

A Carmichael number satisfies a^(n−1) ≡ 1 (mod n) for all a with gcd(a,n)=1, even though n is composite. The first few are: 561, 1105, 1729, 2465, 2821, 6601, …

For n = 561 = 3 × 11 × 17:
  Fermat test with a=2: 2^560 mod 561 = 1  ← looks prime!
  Fermat test with a=5: 5^560 mod 561 = 1  ← looks prime!
  Fermat test with a=10: 10^560 mod 561 = 1 ← looks prime!
  ...every valid witness gives 1 — yet 561 is composite.

Miller-Rabin catches all Carmichael numbers by going one step further: it doesn't just check a^(n−1) ≡ 1, it also examines the sequence of square roots leading up to that result.

4. The Miller-Rabin Insight — Square Roots of 1

The key mathematical fact: in ℤₚ* (where p is prime), the only square roots of 1 are +1 and −1. That is, if x² ≡ 1 (mod p), then x ≡ ±1 (mod p).

This is because x² − 1 = (x−1)(x+1) ≡ 0 (mod p), and since p is prime, p must divide one of the factors — so x ≡ 1 or x ≡ −1 ≡ n−1 (mod n).

Composite numbers can have "non-trivial" square roots of 1 For composite n, x² ≡ 1 (mod n) can have solutions other than ±1. Miller-Rabin exploits this: as we repeatedly square a^d mod n toward a^(n−1) ≡ 1, if we ever see a value that squares to 1 but is itself neither 1 nor n−1, we've found a non-trivial square root — proving n is composite!

5. Decomposing n−1 = 2ˢ · d

Since n is odd (all even numbers > 2 are composite), n−1 is even. We write it as n−1 = 2ˢ · d where d is odd, by factoring out all powers of 2.

DECOMPOSE(n):
    d = n - 1
    s = 0
    while d is even:
        d = d / 2
        s = s + 1
    return (s, d)
    // Result: n-1 = 2^s * d, where d is odd
Example: n = 221
n−1 = 220 = 4 × 55 = 2² × 55
So s = 2, d = 55. We'll compute a^55 mod 221, then square twice.

6. The Squaring Chain — The Core Test

For witness a, compute the squaring chain: a^d, a^(2d), a^(4d), …, a^(2ˢd) = a^(n−1), all mod n.

MILLER-RABIN-WITNESS(n, a, s, d):
    x = modpow(a, d, n)           // compute a^d mod n
    if x == 1 or x == n-1:
        return PROBABLY_PRIME      // passed this round
    for r = 1 to s-1:
        x = x*x mod n              // square: a^(2^r * d) mod n
        if x == n-1:
            return PROBABLY_PRIME  // found -1, chain is valid
    return COMPOSITE               // reached 1 without seeing -1 first
Reading the Chain — 3 Cases
• a^d ≡ 1 (mod n): Passes immediately — consistent with primality.
• Some a^(2ʳd) ≡ −1 (mod n): Found the expected square root of 1 — passes.
• Chain reaches 1 without ever seeing −1: Found a non-trivial √1 — n is COMPOSITE.

8. Modular Exponentiation — Square and Multiply

Computing a^d mod n naïvely would require up to d multiplications — infeasible for large d. The square-and-multiply algorithm does it in O(log d) steps:

MODPOW(base, exp, mod):
    result = 1
    base = base mod mod
    while exp > 0:
        if exp is odd:
            result = result * base mod mod  // multiply
        exp = exp >> 1                       // shift right (halve)
        base = base * base mod mod           // square
    return result

For a 2048-bit exponent, this takes about 2048 squarings and ~1024 multiplications — very fast even for huge numbers.

9. Extended Euclidean Algorithm — Computing d = e⁻¹ mod φ(n)

In RSA, after choosing p and q we compute φ(n) = (p−1)(q−1). The public exponent e must satisfy gcd(e, φ(n)) = 1. The private exponent d is the modular inverse of e modulo φ(n):

EXTENDED-EUCLID(a, b):
    if b == 0: return (g=a, x=1, y=0)
    (g, x1, y1) = EXTENDED-EUCLID(b, a mod b)
    x = y1
    y = x1 - floor(a/b)*y1
    return (g, x, y)

// For RSA: set a = e, b = φ(n)
// If gcd(e, φ(n)) = 1, then x*e + y*φ(n) = 1
// Therefore x ≡ e⁻¹ (mod φ(n)) and d = x mod φ(n)
What the tool shows: The RSA panel displays gcd(e, φ(n)), the Euclidean remainder chain, and the computed inverse d so intermediate number-theoretic steps are visible.

10. Why RSA Decryption Works — m^(ed) ≡ m (mod n)

RSA chooses d so that ed ≡ 1 (mod φ(n)), meaning ed = 1 + kφ(n) for some integer k.

For messages coprime to n, Euler's theorem gives m^φ(n) ≡ 1 (mod n). Then:

m^(ed) = m^(1 + kφ(n))
       = m * (m^φ(n))^k
       ≡ m * 1^k
       ≡ m (mod n)

Since ciphertext is c = m^e mod n, decryption computes c^d ≡ m^(ed) ≡ m (mod n), recovering the original message.
